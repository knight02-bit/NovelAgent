## 二、架构与存储层建议

### 1. KG 与事件存储：强烈建议图数据库优先

你现在用 MemPalace (ChromaDB + SQLite) 做 KG / 语义搜索，这是“上手快”的选项，但**事件本身就是图**（DAG），叙事 KG 也是图，用专门图数据库会舒服得多。

**可以考虑：**

- 事件存储 + KG 用 Neo4j 或类似图数据库
  - EventNode -> 节点
  - `CharacterStateChanged` / `RelationChanged` -> 边 + 属性变化
  - 合并分支时的“差异比较”“冲突检测”，用 Cypher 一条查询就能列出“这个角色在 A 分支和 B 分支的不同属性”。

**组合方式：**

- 向量检索（语义搜索）仍然用 ChromaDB / Qdrant 做 IR
- 结构化关系 & 事件因果，用图数据库

这样 MemPalace 这层就变成“**向量索引 + 分层记忆策略**”，底层实体真实落在图数据库。

如果你先不想引入 Neo4j，也可以先按你文档来，但：

- `EventNode` / 实体关系这块的接口风格要尽量偏“图式”（neighbor 查询、path 查询），未来迁移到图数据库时不会太痛。

### 2. L0-L3 栈：补一个“摘要节点”和“指针”

你已经有 L0-L3 和升降级策略，但落地时有两个关键点：

1. **每个实体需要一个“摘要实体节点”**（summary node）
   - 当角色从 L1 降到 L2 时，不是只存几行文本，而是：
     - KG 里为它挂一个 `summary_of -> character` 的节点
     - 节点上缓存最近状态（关系、重大事件）
   - 回到 L1 时：
     - 优先读这个 summary node 构造 prompt，而不是从一堆旧事件里再总结一遍
2. **“整卷摘要化归档”**要和世界观版本打通
   - 卷结束时，生成“卷级世界观快照 + 卷级摘要节点”：
     - WorldState(vN) 节点挂一个 `has_summary -> VolumeN`
     - 后面查询某角色在第 N 卷末的状态，只需要 1~2 跳查询

这样可以把你设计中的“世界观版本管理”和“记忆栈”硬连起来，避免后期两套系统各玩各的。

## 三、Agent & LLM 使用相关建议

### 1. LLM Provider：场景 -> 档位 -> 模型 已经很好，可以再加两条

在你现有基础上再补两个可观测维度，会对“生产可控性”很重要：

1. **为每次 generate 记录“上下文 token 构成”**：
   - 比如：`{system: 10%, L0: 20%, L1: 30%, dynamic: 40%}`
   - 存进 `SystemEvent.metadata["prompt_profile"]` 中
   - GUI 可以画出“这一章 / 这一卷的 L0/L1 使用程度”曲线，有助于调参数
2. **在场景 routing 再多一个“节奏控制”维度**：
   - scene 不仅仅是 `"narrative" / "analysis" / "filter"`
   - 还可以有 `"fast_pace" / "slow_pace" / "dialogue_heavy"`
   - Provider 自动把节奏类型映射到不同的采样参数（温度 / top_p / max_tokens）

这样“节奏引擎”和 LLM Provider 有一个非常自然的连接，不需要每次在 prompt 里啰嗦“节奏加快一点”之类。

### 2. Prompt Cache：抽象成“Prompt Segment”对象

你已经做了 Part1~Part4 分层，建议在实现时不要只在 Provider 里做字符串切片，而是显式建一个“小 DSL”：

python复制

```
@dataclass class PromptSegment:    id: str          # e.g. "system_core", "world_v3", "chapter_12"    content: str    cacheable: bool    ttl_seconds: int | None
```

- Agent 构建 prompt 时是拼 `List[PromptSegment]`
- Provider 把连续 `cacheable=True` 的前缀打成缓存块，适配各厂商 API 的 cache 控制字段
- 后面如果想引入“语义层级缓存”（比如同一世界观不同章节也复用一段规则），这层抽象会特别有用。

### 3. Brainstorming Service：从“多模型并行”改成“多阶段漏斗”

你设计是：多模型并行生成 -> Consistency Filter -> GUI 选择。

建议升级为**三阶段漏斗**（但代码上仍可简单实现）：

1. **扩散阶段**：1~2 个高温度模型，大量生成（N 条）
2. **收敛阶段（低成本模型）**：
   - 对 N 条 candidate 做：
     - KG 一致性基本过滤
     - 去重 / 合并相似思路（向量相似度 + 简单规则）
   - 输出 K 条“代表性路线”
3. **打磨阶段（高质量模型）**：
   - 对这 K 条，逐条调用高档位模型 rewrite / expand
   - 这一步也可以携带“伏笔回收 / 结构节拍要求”

好处：

- 你不需要维护 5~6 种模型同时工作的复杂度
- 成本从“一次脑暴 5 * 高 token”降为“先便宜粗筛+少量精调”
- 和 Consistency Filter 逻辑天然衔接。

## 四、GUI / 交互上的创意与优化

这里给几个“作者向”的具体交互点，可以明显拉开和一般 AI 写作工具的差距。

### 1. “故事卷轴”视图（Story Roll）

在你已有的分支 DAG 基础上，加一个“横向时间卷轴”视图：

- 横轴：时间 / 章节
- 纵向：不同分支（类似地铁线路图 / 命运石之门世界线）
- 鼠标滑动：
  - 中部正文区跳转到对应章节
  - 右侧 KG 区自动切换到当时世界观版本
  - 上方小条显示“这段时间伏笔数量 / 未回收伏笔数”

对写长篇的作者，这个会非常直观，也比单纯的 DAG 树节点好理解。

### 2. “设定合法性灯”（世界观一致性小灯）

在正文编辑区域，每一段 / 每个事件后面显示一个“小灯”：

- 绿色：完全符合当前世界观 + KG
- 黄色：存在轻微冲突（比如角色情绪反应和既有性格不完全吻合）
- 红色：严重冲突（设定推翻 / 时间线不可能）

你已经有 Consistency Filter，可以直接复用结果挂到 UI 上。
这比弹窗提示友好很多，也会成为作者在创作时直观看状态的指示灯。

### 3. “伏笔追踪面板”的反向用法：从伏笔倒推候选场景

现在设计中伏笔追踪是“埋伏笔 -> 记录 -> 检测是否回收”，可以再加一条逆向路径：

- 作者在伏笔面板选中一个还没回收的伏笔
- 点击“从这里脑暴回收方案”
- Brainstorming Service 以“必须显式触及该伏笔的 payoff”为硬约束生成候选

这样 Brainstorming 就不仅是“往前推故事”，还是“从伏笔倒推事件”，体验会明显不一样。
